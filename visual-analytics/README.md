# EIF  – Behavioral Anomaly Detection for Mule-Hunter

EIF implements **Extended Isolation Forest (EIF)** for detecting behavioral anomalies in financial transaction data within the **Mule-Hunter fraud detection system**.

The model identifies suspicious accounts by analyzing transaction behavior patterns and assigning an **anomaly score**.  
To improve transparency and interpretability, **SHAP (SHapley Additive exPlanations)** is used to explain which behavioral features contributed to the anomaly detection.

---

# Overview

Fraud detection systems face several challenges:

- Highly **imbalanced datasets**
- **Evolving fraud strategies**
- Limited labeled fraud data

To address this, Mule-Hunter uses **Extended Isolation Forest**, an **unsupervised anomaly detection algorithm** that isolates abnormal behavioral patterns in transaction data.

EIF helps identify accounts showing suspicious activity such as:

- abnormal transaction velocity
- high fan-out transfers
- inconsistent device usage
- irregular transaction timing
- unusual transaction amount patterns

---

# Architecture in Mule-Hunter

The EIF model serves as the **behavioral anomaly detection layer** in the fraud detection pipeline.

```
Transaction Data
       ↓
Behavior Feature Engineering
       ↓
Extended Isolation Forest (EIF)
       ↓
Anomaly Score
       ↓
SHAP Explainability
       ↓
Fraud Risk Engine
```

EIF detects anomalous behavior, while SHAP provides **interpretability for model decisions**.

---

# Explainability with SHAP

To understand why an account is flagged as anomalous, the system uses **SHAP**.

SHAP helps explain:

- which features influenced the anomaly score
- the contribution of each behavioral signal
- why the model marked an account as suspicious

Example explanation:

| Feature | Impact |
|--------|--------|
| tx_velocity_7d | High transaction frequency increased anomaly score |
| fan_out_ratio | Large number of outgoing transfers |
| amount_entropy | Irregular transaction amounts |

This makes the system more **transparent and auditable**.

# Behavioral Features Used

The model analyzes behavioral signals derived from transaction activity.

| Feature | Description |
|--------|-------------|
| tx_velocity_7d | transaction frequency over 7 days |
| fan_out_ratio | ratio of outgoing transfers |
| amount_entropy | randomness of transaction amounts |
| device_consistency | device usage stability |
| addr_entropy | diversity of IP/location |
| d_gap_mean | average time gap between transactions |
| in_out_ratio | incoming vs outgoing transfers |
| reciprocity_score | bidirectional transfer behavior |

These features help capture **financial behavior patterns** associated with mule accounts.

# Why Extended Isolation Forest?

EIF was chosen because it:

- works well with **imbalanced fraud datasets**
- detects **unknown fraud patterns**
- scales to **large transaction datasets**
- requires **no labeled fraud data**

These properties make it ideal for **behavioral fraud detection systems**.

---

# Integration with Mule-Hunter

The anomaly score generated by EIF is used by Mule-Hunter to:

- flag suspicious accounts
- support graph-based fraud detection
- assist the risk scoring engine

Combined with other modules, this creates a **multi-layer fraud detection system**.
